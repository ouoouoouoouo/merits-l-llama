# MERITS-L with Llama-3.1-8B Text Encoder

An experimental extension of the MERITS-L pipeline for multimodal emotion recognition in conversations.

This repository focuses on replacing the original RoBERTa-large text branch with **Meta-Llama-3.1-8B**, using LoRA or full-parameter fine-tuning, while retaining the dialogue-context and multimodal fusion stages used by MERITS-L.

## Current scope

The currently implemented pipeline is:

```text
Text branch:
Llama-3.1-8B
    ↓
Stage I utterance-level emotion classification
    ↓
4096-dimensional utterance features
    ↓
Stage II Bi-GRU + self-attention
    ↓
2048-dimensional dialogue-aware text features

Audio branch:
WavLM-Base + CARE
    ↓
Audio Stage II features reused from merits-l-text
    ↓
256-dimensional dialogue-aware audio features

Multimodal branch:
Text/audio co-attention
    ↓
Stage III emotion classification
```

> **Important**
>
> The current Stage III implementation uses the original **WavLM-Base + CARE** audio features generated by the sibling `merits-l-text` repository.
>
> The proposed **WavLM-Large + CARE-Large** audio upgrade and the use of Llama embeddings as the CARE semantic-alignment target are planned extensions and are not included in the current repository.

## Research objective

The main objective is to study whether replacing RoBERTa-large with a larger decoder-only language model improves speech emotion recognition on the four-class IEMOCAP setting:

* angry
* happy, including excited
* sad
* neutral

The repository supports three Llama training strategies:

1. Direct LoRA fine-tuning on IEMOCAP
2. MSP-Podcast pretraining followed by IEMOCAP fine-tuning
3. Full-parameter fine-tuning

The sibling `merits-l-text` repository contains the RoBERTa-based baseline and the original WavLM-Base + CARE audio pipeline.

## Differences from the baseline

| Component                   | Baseline                   | This repository                        |
| --------------------------- | -------------------------- | -------------------------------------- |
| Text encoder                | RoBERTa-large              | Meta-Llama-3.1-8B                      |
| Encoder hidden size         | 1024                       | 4096                                   |
| Fine-tuning method          | Standard fine-tuning       | LoRA or full fine-tuning               |
| Utterance pooling           | CLS or mean pooling        | Mean or last-token pooling             |
| Stage II input              | 1024-dimensional           | 4096-dimensional                       |
| Stage II Bi-GRU hidden size | 512                        | 1024                                   |
| Stage II output             | 1024-dimensional           | 2048-dimensional                       |
| Stage III audio branch      | WavLM-Base + CARE          | WavLM-Base + CARE reused from baseline |
| Stage III fusion            | Bidirectional co-attention | Bidirectional co-attention             |

Because the Stage II GRU size is also increased, the current comparison does not isolate the text encoder as the only experimental variable.

## Three-stage architecture

### Stage I: utterance-level text classification

Stage I fine-tunes Llama-3.1-8B for four-class emotion recognition at the utterance level.

The default configuration uses LoRA adapters on:

```text
q_proj
v_proj
```

The non-padding token representations are mean-pooled into one 4096-dimensional utterance representation.

Available variants:

* Direct IEMOCAP LoRA fine-tuning
* MSP-Podcast-pretrained LoRA fine-tuning
* Full-parameter fine-tuning

### Stage II: dialogue-context modeling

Stage II takes the precomputed 4096-dimensional Llama representation of each utterance and applies:

```text
Bi-GRU
    ↓
Multi-head self-attention
    ↓
Utterance-level emotion classifier
```

The resulting dialogue-aware representation has 2048 dimensions.

### Stage III: multimodal fusion

Stage III combines:

* 2048-dimensional Llama text features
* 256-dimensional WavLM-Base + CARE audio features

The fusion model applies bidirectional co-attention:

```text
Text attends to audio
Audio attends to text
        ↓
Residual connection and layer normalization
        ↓
Feature concatenation
        ↓
Emotion classifier
```

The audio Stage II hidden features must currently be generated or copied from `merits-l-text`.

## Prerequisites

### Model access

Access to the following gated Hugging Face model is required:

```text
meta-llama/Meta-Llama-3.1-8B
```

The Hugging Face account used on the training machine must have permission to download the model.

### IEMOCAP manifests

This repository expects the IEMOCAP manifests at:

```text
data/manifests/iemocap/
├── train.csv
├── val.csv
└── test.csv
```

The manifests can be generated using the preprocessing pipeline in `merits-l-text`.

### Stage III audio features

Stage III additionally requires:

```text
data/cache/iemocap_audio_stage2_utt.pt
```

This file is generated by the WavLM-Base + CARE audio pipeline in `merits-l-text`.

## Installation

For CUDA 12.4 and RTX 4090-class GPUs:

```bash
python -m venv .venv
source .venv/bin/activate
pip install --upgrade pip
pip install -r requirements.txt
```

For NVIDIA Blackwell GPUs, use the separately pinned environment:

```bash
pip install -r requirements-blackwell.txt
```

The dependency files target different CUDA and PyTorch combinations. Do not install both files into the same environment without checking version compatibility.

## Quick start

The following commands describe the standard direct-IEMOCAP LoRA pipeline.

### 1. Train Stage I

```bash
python -m src.train \
    --config configs/iemocap_text_llama_stage1.yaml
```

The best LoRA adapter is expected under:

```text
outputs/iemocap_text_llama_stage1/best/
```

### 2. Extract Llama utterance features

```bash
python -m scripts.extract_text_features_llama \
    --config configs/iemocap_text_llama_stage1.yaml \
    --lora-ckpt outputs/iemocap_text_llama_stage1/best \
    --out-pt data/cache/iemocap_llama_features.pt
```

The output is a dictionary of:

```text
utterance ID → tensor with shape (4096,)
```

### 3. Train Stage II

```bash
python -m src.train_stage2 \
    --config configs/iemocap_text_llama_stage2.yaml
```

### 4. Extract Stage II dialogue-aware text features

```bash
python -m scripts.extract_stage2_utt_hidden \
    --config configs/iemocap_text_llama_stage2.yaml \
    --stage2-ckpt outputs/iemocap_text_llama_stage2/best/stage2.pt \
    --out-pt data/cache/iemocap_text_llama_stage2_utt.pt
```

The output representation has shape:

```text
utterance ID → tensor with shape (2048,)
```

### 5. Prepare the audio features

Copy the audio Stage II features produced by `merits-l-text`:

```bash
mkdir -p data/cache
cp /path/to/merits-l-text/data/cache/iemocap_audio_stage2_utt.pt \
   data/cache/iemocap_audio_stage2_utt.pt
```

### 6. Train Stage III

```bash
python -m src.train_stage3 \
    --config configs/iemocap_stage3_llama.yaml
```

## Results

All numbers are IEMOCAP 4-class weighted F1 on the session-based test split
(1241 utterances). Stage I and Stage II are single seed=42 test scores.
Stage III fusion reports **mean ± std over 5 random seeds** and the
**best-of-5** seed value. The audio branch (WavLM-Base + CARE) is identical
across all rows; validation checkpoint-selection metric is weighted F1.

| Experiment                       | Stage I wF1 | Stage II wF1 | Stage III wF1 (mean ± std) | Stage III best |
| -------------------------------- | ----------: | -----------: | -------------------------: | -------------: |
| **Direct IEMOCAP LoRA** ★        |  **0.6659** |   **0.8015** |     **0.8567 ± 0.0139**    | **0.8746** ★   |
| MSP-Podcast pretraining + LoRA   |      0.7038 |       0.7956 |         0.8550 ± 0.0072    |         0.8640 |
| Full fine-tuning (with pretrain) |      0.7230 |       0.8178 |         0.8294 ± 0.0137    |         0.8529 |

Reference: the MERITS-L paper (single reported value) is **0.8648** on this
setting. The best-of-5 seed of the Direct IEMOCAP LoRA route exceeds this
by +0.98 percentage points; the MSP-Podcast-pretrained LoRA route effectively
matches it (-0.08 pp). Full fine-tuning improves the uni-modal text stages
(Stage I 0.7230, Stage II 0.8178) but reduces the multimodal fusion score,
suggesting that parameter-efficient adaptation preserves features that are
more complementary to the audio branch under co-attention.

Do not compare a best single run against a multi-seed mean without explicitly
identifying the difference.

## Hardware notes

Llama-3.1-8B requires substantially more GPU memory than RoBERTa-large.

The actual memory requirement depends on:

* LoRA rank
* sequence length
* batch size
* gradient accumulation
* precision
* optimizer
* gradient checkpointing
* full fine-tuning versus parameter-efficient fine-tuning

The default LoRA configuration is intended for high-memory consumer or data-center GPUs. Full-parameter fine-tuning requires significantly more memory and should not be assumed to fit on the same hardware as the LoRA configuration.

Exact VRAM figures should only be added after recording them from a reproducible run.

## Recommended ablations

To separate the effect of the text encoder from the effect of increased Stage II capacity, the following controlled experiments are recommended:

1. RoBERTa and Llama projected to the same feature dimension
2. Identical Stage II GRU hidden sizes for both encoders
3. Frozen Llama features versus LoRA-adapted Llama features
4. Mean pooling versus last-token pooling
5. Direct IEMOCAP training versus MSP-Podcast pretraining
6. Baseline audio versus CARE-Large audio
7. Single-modality text results versus multimodal fusion results
